Auto Disk Cloning (autoreplicated_top) Flow

In the Xcelium Single-Core Replicated Build Flow, you had to manually find the replicated_top partition. 

But now, the tool is enhanced to find the replicated partition(s) automatically, without any human intervention. This is enabled under the -autoreplicated_top option.  The -autoreplicated_top option does not need any argument. The diagram below depicts the auto disk cloning (autoreplicated_top) flow:


In the image above, an extra elaboration phase "Automatic Finding of Replicated module" is added as compared to the manual disk cloning flow.

This -autoreplicated_top option forks off two xrun commands serially:

  1. xrun does the compilation and partitioning for finding the replicated module.
  2. xrun does the cross-partition analysis, replicated partition build and then cloning with final build model.

In the very first scratch run, there is some extra time taken since the partitioner needs to be run. Since the overall structure of the design does not change much, with that assumption, a new flow is added with -MSIE_USEPARTITIONFILE. Using this option with -autoreplicated_top, you can skip the partitioner run if you have a partition file available. By default, the partition file is generated inside xcelium.d/run.d/partition.txt.


In the second scratch run, you can copy the xcelium.d/run.d/partition.txt and use it as:

xrun -autoreplicated_top -msie_usepartitionfile ./partition.txt <OTHER_COMMAND_LINES>

This run uses the available partition file and skips the partitioner run. This can save some amount of time. 

The autoreplicated_top option has:

  • Single xrun invocation flow where all partitions are elaborated with single CPU core
  • Ease of use: Use the -autoreplicated_top option with Monolithic elaboration
  • Replicated module name that become partial partition


XMSIE_MULTIPLE_REPLICATED_TOPS

There are multiple kinds of replication in a design.

In a scenario where there is one replication block that has multiple instantiations and whose cumulative weight is more than 50%, -autoreplicated _top is adequate.

There are other testcases where distributed replications are seen, owing to multiple replicated blocks. For those distributed replication design, use -XMSIE_MULTIPLE_REPLICATED_TOPS option (along with -autoreplicated_top), which enables the dumping of multiple replicated tops. The maximum limit for number of replicated tops to be dumped is set to 100.


The following image illustrates the Xcelium Single Core Auto-replicated Build Flow with XMSIE_MULTIPLE_REPLICATED_TOPS:

The usage is given below:

xrun -autoreplicated_top -XMSIE_MULTIPLE_REPLICATED_TOPS <OTHER_CMD_LINES>

To debug, enable the partitioner verbosity with the -xmsie_verbose option.
