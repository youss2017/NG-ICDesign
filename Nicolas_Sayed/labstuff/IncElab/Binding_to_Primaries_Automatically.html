Multi-Snapshot Incremental Elaboration
Product Version 23.09, September 2023

Binding to Primaries Automatically

Single-step invocation MSIE flow has binding similar to the Monolithic flow and so no extra option is required. The tool does the binding under the hood.

Specifying the -primbind  option on the command line includes primary snapshots in the final elaboration automatically in multi-step and direct invocation MSIE flows. This option removes the requirement of having to specify all primary snapshots at elaboration time and allows for instance-specific partition boundaries. It also adds greater partitioning flexibility by using standard binding rules to decide which instances bind to which primaries.

The xrun and xmelab flows both support the -primbind option in multi-step and direct invocation MSIE flows.

For example:

xmelab tb primbind

During final elaboration, this option (xmelab -primbind or xrun -primbind) includes primary snapshots when building the simulation snapshot.

Binding Primaries to Instances

When using the -primbind option, the cell name of the primary snapshot is significant. If the elaborator is searching for a match to a particular cell name, whether it comes from a configuration specification or from the name used in a module or component instance, it finds only those primary snapshots with a matching cell name. The same holds true in cases where a library name or view name is also specified. This is similar to how VST and AST design units are found.

When a primary snapshot has been found as the design unit that should be bound to an instance, there is still the matter of determining which top-level module or entity in that primary’s hierarchy should be bound to the given instance. Primary snapshots may be built with multiple top-levels; however, when using the -primbind option, only one top-level unit can be bound to the specified instance during final elaboration. The elaborator attempts to determine which top-level to bind according to the following rules:

  • If there is only one top-level module or entity, then that unit is bound to the instance.
  • If there are multiple top-level modules or entities available, then the one that matches the cell name of the primary snapshot is bound to the instance.
  • If there are multiple top-level modules or entities and none of them match the cell name of the primary snapshot, or there are no top-level modules or entities, then the elaborator generates an error.

In those cases where there are remaining top-level units in a primary partition, these become top-level design units of the incremental partition during the final elaboration and cannot be used for binding to other instances.

When using the -primsnap option, the elaborator follows these binding rules:

For Verilog, name mapping is used to bind primaries to instances. The name of the module in the instantiation statement is compared with the name of each top-level module in the primary. The first one whose name matches is bound.

For VHDL, there are two ways of binding an instance to a primary snapshot:

  • By name mapping. The name of the component must match the name of a top-level entity in the primary.
  • By using a binding indication that specifies the binding of this particular component instantiation to the top-level design unit of the primary.

When -primbind and -primsnap are used together, the primaries given with -primsnap are searched first using the -primsnap rules above. If that does not result in a match, then the -primbind rules is used.

It is recommended to use -name/-snapshot option along with -mkprimsnap option on primary snapshot creation with xrun.

At the time of incremental elaboration, use -primname <PRM_NAME>@.<PRIM_LIB_DIR_PATH>.

The -primname option specifies which previous xrun command created the primary snapshot.

For example:

xrun -primname <name>@<directory> [other_options] <source_files>

xmelab -primname <name>@<directory> [other_options] [lib.]cell[:view]

The name argument can be:

  • The argument to the -snapshot option specified on the xrun command line used to create the primary snapshot.
  • The argument to the -name option specified on the xrun command line used to specify the top-level design unit when creating the primary snapshot.
  • The argument to the -xmuid option given to the first xrun command. 

If -xmuid is used in addition to the -snapshot option, specify the argument used for -xmuid.

  • xrun

If the -snapshot, -name, or -xmuid options are not used, specify xrun as the argument to the -primname option.

If the primary snapshot was built in another directory or with the -xmlibdirname option, the directory must also be specified. This directory can also be specified with the -primlibdir option if only one -primname option is used.

Merging Multiple Top Modules Automatically

During incremental elaboration, if the model ends up with multiple top-level modules that all have the same name, but comes from different partitions, the simulator attempts to merge the duplicate top modules together automatically. If the simulator cannot merge these duplicate tops, then an error is genrerated. In order to suppress this error, use the -msietopsok option on the command line.
