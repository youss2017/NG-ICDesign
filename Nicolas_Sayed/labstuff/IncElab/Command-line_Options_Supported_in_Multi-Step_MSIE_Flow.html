Command-line Options Supported in Multi-Step MSIE Flow

The following options are supported in the Muli-step MSIE flow.

OptionDescription

-bbox_create

When elaborating a primary snapshot, use xrun -bbox_create option to create a self-sufficient IP model that acts as a black box. This black box primarily contains the snapshot, and includes all dependent files and work libraries in the designated directory. If the directory does not already exist, then the elaborator creates it automatically.

% xrun -mkprimsnap -incrbind top -bbox_create ./tmp/box1 ip1.v
% xrun -mkprimsnap -incrbind top2 -bbox_create ./tmp/box2 ip2.v

A package that is elaborated in a black box primary is not shared with the same package in the incremental. Normally, the incremental re-uses package instances that it finds in primary snapshots and shares any global objects found in that package. However, when elaborating a primary with -bbox_create, the software elaborates another instance of the package and this can lead to unexpected simulation results.

The -cds_implicit_tmpdir and -bbox_create options are not compatible. Do not use both options together on the command line.

The specified directory is automatically created during primary elaboration if it does not already exist.

-bbox_link

When elaborating the final/incremental snapshot, use this option to attach a self-sufficient IP model from a particular directory without having to re-elaborate the design. If there are multiple IP models (or black boxes), then you must specify multiple -bbox_link options corresponding to each IP model. This option specifies the bbox directory where the primary snapshot is created.

For example, the -bbox_create option is used to elaborate two self-sufficient IP models. The blackboxed IP models IP1 and IP2 are available to the SOC integrator in the ./tmp/box1 and ./tmp/box2 directories. The SOC design is in the incremental partition. Using the -bbox_link option, the SOC integrator can attach IP1 and IP2 to the design, soc.v.

The following shows how to integrate the black-boxed IP models into the SOC design, soc.v:

% xmvlog soc.v
% xmelab -primbind -bbox_link ./tmp/box1 \
 -bbox_link ./tmp/box2 soc

Or use xrun, as shown:

% xrun -primname ip1 -bbox_link ./tmp/box1 -primname ip2 \
  -bbox_link ./tmp/box2 soc.v

cds_implicit_tmpdir (Directory for Holding Temporary Libraries)

When using the -cds_implicit_tmpdir option to build a primary snapshot, you must elaborate the simulation snapshot with a corresponding -cds_alternate_tmpdir option on the command line when creating the final model. Regardless of whether -primsnap, -primname, or -primbind is used to include a primary in the final model, the primary is not visible without it. For example:

  • Direct invocation mode

    # Create a primary snapshot
    % xmelab -mkprimsnap tb_top -cds_implicit_tmpdir dir1

    # Create the simulation snapshot
    % xmelab tb_top -primbind -cds_alternate_tmpdir dir1

  • Multi-step xrun invocation mode

    # Create a primary snapshot using xrun
    % xrun -mkprimsnap -incrbind test tb_top.v -snapshot tb_top \
    -cds_implicit_tmpdir dir1

    # Create the simulation snapshot
    % xrun -primname tb_top test.v -cds_alternate_tmpdir dir1

    For single-step xrun invocation mode, the software takes care of managing the snapshots during primary and incremental elaboration. Ensure not to mix single-step xrun with either multi-step xrun or direct invocation mode when using MSIE.

    The cds_implicit_tmpdir is not a recommended option. Use of bbox_create option is recommended. While using -mkprimsnap option, we use -name in place of -snapshot option.

-forceprimupdate

If a primary snapshot has changed, it is not updated if the incremental snapshot is being built for the first time. The current flow assumes that the primary snapshot is up-to-date. However, this assumption might not be true.

The -forceprimupdate option allows the primary snapshot to be checked (and updated if needed) even when the incremental snapshot is being built for the first time.

-genhref filename

Generates a hierarchical permission file (href file).

When specifying a filename, you can use a simple name to generate the href file in the current working directory or a direct path to generate the href file in some other location.

For example:

-genhref href.txt

or:

-genhref test/href.txt

See Hierarchical References.
-gentypehref Generates type based href file.

-href filename
-href "@href_permission_specification"

Specifies hierarchical reference permissions, which give the incremental partition the ability to refer to certain declarations in the primary with a hierarchical reference.

The argument to the -href option can be:

  • A hierarchical reference permission file. This is a text file that specifies reference permissions to objects in the primary snapshot.
  • A permission specification of the form:

    "@absolute_path permission"

    For example:-href "@dut.r1 *"

See Hierarchical References.

-incrbind module_name

Identifies a module whose instances shall be left unbound in the primary partition so that they can be bound to other partitions in the final model.

There may be times when one or more module instances within a primary snapshot are left unelaborated. Another partition, either primary or incremental, contains these instances. When elaborating the primary snapshot, use the -incrbind command-line option to identify a module or program block to bind to another partition. This pertains to all instances of the specified module or program block.

The specified module cannot be a SystemVerilog interface.

When using the -genhref flow, use the -incrtop option in place of -incrbind. For backward compatibility, use the -top option in place of -incrbind.
-incrhasvhdl

VHDL is 9 valued logic language while Verilog is 4 valued logic.

For correct simulation results in mixed-language testcases there must be a proper mapping between VHDL and Verilog design across MSIE boundaries. To achieve this the primary (with Verilog/SystemVerilog content) must be updated with -INCRHASVHDL option so that the incremental or other primaries can have VHDL.

-incrpath path
-incrpath top_level_unit@path

When building a primary partition, specifies the path(s) to this partition's top-level unit(s) within the overall design hierarchy. This option is used to inform the elaborator when it needs to resolve hierarchical names that start from the top of the overall design hierarchy.  Examples of such names are:

  • Hierarchical references in the HDL that originate from and terminate in the primary partition
  • Hierarchical access files, timing files, X-propagation files and SDF files that apply to the overall design
  • Path names of parameters in -defparam options

The argument to the -incrpath option can be any full path, for example:

  • A hierarchical path name

  • A hierarchical path name given to the -defparam option or defparam statement

    If the primary partition being built with this option has multiple top-level units, this second form of the argument must be used.

  • A name of a top-level unit followed by a hierarchical path name for that top-level unit

-lwbbox

The light weight bbox solution is provided under the -lwbbox option. This option needs to be provided on top of -bbox_create and -bbox_link at primary and incremental build.

The light weight bbox solution prevents copying the compilation design units (VSTs) into each primary's black-boxed space. This ensures a lower disk space and therefore, lower incremental peak memory and better CPU utilization (less I/O). The -lwbbox option also enables reusing the packages (already elaborated as part of the primary snapshot) in the incremental.

Example – Reuse package from primary

//pkg.sv
package p;
 reg r;
endpackage 
//prim1.sv
module prim1;
import p::*;
endmodule 
//prim2.sv
module prim2;
import p::*;
endmodule 
//incr.sv
module top;
import p::*;
  prim1 prim1();
  prim2 prim2();
endmodule 
// runme_bbox
xrun -mess -mkprimsnap -name prim1 pkg.sv prim1.sv -top prim1 -bbox_create bbox1 -xmlibdirname ./primlib1
xrun -mess -mkprimsnap -name prim2 pkg.sv prim2.sv -top prim2 -bbox_create bbox2 -xmlibdirname ./primlib2
xrun -mess pkg.sv incr.sv -primname prim1@./primlib1 -primname prim2@./primlib2 -bbox_link bbox1 -bbox_link bbox2 
//runme_lwbbox
xrun -mess -mkprimsnap -name prim1 pkg.sv prim1.sv -top prim1 -bbox_create bbox1 -xmlibdirname ./primlib1 -lwbbox
xrun -mess -mkprimsnap -name prim2 pkg.sv prim2.sv -top prim2 -bbox_create bbox2 -xmlibdirname ./primlib2 -lwbbox
xrun -mess incr.sv -primname prim1@./primlib1 -primname prim2@./primlib2 -bbox_link bbox1 -bbox_link bbox2 log -lwbbox

With bbox – package gets re-elaborated in the incremental: 

Loading primary snapshot worklib.prim1:sv .................... Done
Loading primary snapshot worklib.prim2:sv .................... Done
            Top level design units (incremental partition):
                        p
                        top

With lwbbox – package gets re-used in the incremental 

Loading primary snapshot worklib.prim1:sv .................... Done
Loading primary snapshot worklib.prim2:sv .................... Done
            Top level design units (incremental partition):
                        top

-mkprimsnap

Creates a primary snapshot design unit.
-msieunlock cw_intf_optm Aims at optimizing interfaces in multi-step MSIE flow to gain simulation performance.
-noprimupdate In Multi-step MSIE flow, you can use the -noprimupdate option to disable automatic updates to the primary snapshot(s).

-primincrpathok

Suppresses the error message that is generated when an out-of-module reference is resolved using the path information provided to the -incrpath option, and that information does not match the actual path in the incremental partition. This option provides flexibility in constructing custom models.

Without this option, the elaborator generates an INCMSM error.

-primbind

Includes primary snapshots when searching design libraries for units to bind to instances when elaborating the simulation snapshot. Primary snapshots are chosen over modules with the same name.

This option is required on the command line in order to include primary snapshots in a given MSIE model during final elaboration if they are not named explicitly with the -primsnap option, or the -primname option when using xrun.

For xrun, this option works only when specifying the name of the primary top module. If a primary snapshot is configured such that a top module is specified in the primary snapshot but is not the primary top, then the incremental snapshot is unable to find the primary snapshot and the software issues an error.
This option works only if the name of the primary snapshot is the same as the name used in the instantiation or Verilog configuration (or the name of the module if it is instantiated at the top of the hierarchy). Otherwise, the elaborator is unable to find the primary snapshot and the software does one of the following:
  • Re-elaborate the contents of the primary inside the incremental, if the primary's compiled design units are accessible.
  • Generate a binding error, if the primary's compiled design units are not available.

Typically, a primary snapshot needs to have the same name as its top-level module, since that is the name used in instantiations and configurations to refer to the module that is meant to be bound to that instantiation.

To use a primary snapshot built with -bbox_create, the MSIE model must be elaborated with a corresponding -bbox_link option on the command line.

-primhrefupdate

When running either multi-step xrun or direct invocation mode, this option enables automatic re-elaboration of the primary snapshot when objects need additional hierarchical reference permissions during incremental elaboration. The -primhrefupdate option uses the autohref.txt file to update permissions in the primary snapshot and then continues elaborating the incremental snapshot.

See Generating an href File When Elaborating the Simulation Snapshot.

-primlibdir directory

When using separate xrun commands to build the primary and simulation snapshots, this option can specify either:

  • The directory where the first xrun command was run, if the invocation directories are different.
  • The directory specified with the -xmlibdirname option to the first xrun command.

Using the name@directory form of the -primname option is preferred.

See XRUN Invocation.

-primparamsok

Suppresses error messages that are generated when a defparam statement or parameter override in one partition tries to change the value of a parameter or generic in another primary partition.

Once a primary snapshot is elaborated, the value of each parameter and generic in that primary partition is fixed. Overwriting a parameter or generic value during final elaboration is not allowed. By default, the elaborator generates an error if something in the final elaboration attempts to change the value of a parameter or generic in a primary partition. Use the -primparamsok option to suppress these error messages.

This option is not recommended unless you have verified that the parameter value is not play a role in primary snapshot. 

-primsnap snapshot_name

When running in multi-step invocation mode, this option specifies a primary snapshot to use with the simulation snapshot. When binding instances to design units, the top-level units of primaries named with the -primsnap option are considered first, ahead of any other binding rule.

This option is now deprecated in favor of the -primbind option.

The specified snapshot must have been elaborated previously with the -mkprimsnap or -incrbind options. Use the -primsnap option with the -primbind option to override the binding decisions made by the normal binding rules.

-primsnap is not a recommended option. It is recommended to use -primname option.

-primname name[@directory]

Indicates the primary snapshot when using the xrun flow.

When using MSIE, separate xrun commands are required to build one or more primary snapshots and the final simulation snapshot. This option indicates which xrun commands created the primary snapshots.

The name argument can be:

  • The argument to the -snapshot option specified on the xrun command line used to create the primary snapshot
  • The argument to the -name option specified on the xrun command line used to specify the top-level design unit when creating the primary snapshot
  • The argument to the -xmuid option given to the first xrun command
    If -xmuid is used in addition to the -snapshot option, specify the argument used for -xmuid

  • xrun
    If the -snapshot, -name, or -xmuid options are not used, specify xrun as the argument to the -primname option.

    -snapshot and -name are synonyms for the same option.

    If the primary snapshot was built in another directory or with the -xmlibdirname option, the directory must also be specified. This directory can also be specified with the -primlibdir option if only one -primname option is used.

-primvhdlcompat

Generates a primary snapshot that is compatible with VHDL.

When building a primary snapshot that does not contain any VHDL, the -primvhdlcompat option is required if the intent is to use that primary snapshot in a model that does contain some VHDL.

If this option is missing, and you attempt to generate an incremental snapshot for a model that contains primary snapshot with some VHDL, the elaborator halts after generating a DYNVPH error. To resolve the error, you need to rebuild the primary snapshot by adding the missing -primvhdlcompat option to the command line.
