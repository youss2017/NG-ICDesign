Hierarchical References

By default, when a primary partition is elaborated, the elaborator has no knowledge of the rest of the design.

Hierarchical names that would resolve to declarations in another partition result in an unresolved hierarchical name error unless you map the names to real object paths using the -genhref option.

When elaborating a primary partition that is not at the top of the overall design hierarchy, a hierarchical reference that uses a full path name starting at the top of the hierarchy also gives an unresolved name error, even though the name refers to an object in the primary snapshot. However, you can use the -incrpath option to specify the path(s) to the primary partition and overcome this problem.

Hierarchical references in the incremental partition do not have this problem because the whole design hierarchy is known when the simulation snapshot is being elaborated. However, for performance reasons, each object in any primary partition that is the target of a hierarchical reference from the incremental partition must be given special permission when the primary snapshot is elaborated. If the necessary permissions have not been granted, the elaborator generates an error.
