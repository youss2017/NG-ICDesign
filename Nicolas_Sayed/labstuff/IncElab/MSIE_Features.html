MSIE Features

MSIE flow provides many features. It helps you build the complete MSIE model using a single xrun command, provides the auto partitioning feature that automatically creates optimum partitions in the design, and so on. This topic lists details of the features available with the MSIE flow:

  • Auto-MSIE Partitioner in Single-Step Invocation MSIE Flow: Auto partitioning solution in MSIE (also commonly referred to as Auto-MSIE) is provided for automatically creating optimum partitions in the design. It divides the entire test case into multiple primary partitions. To set the name of the partition file, use the -genpartition <partition-file> option, where partition-file is an absolute/relative path. To force the number of partitions, use the -xmsienumpartition <num> option. By default, the design is divided into 12 primaries.
  • Binding to Primaries Automatically: Single-step invocation MSIE flow has binding similar to the monolithic flow and so no extra option is required. The tool does the binding under the hood. Specifying the -primbind  option on the command line includes primary snapshots in the final elaboration automatically in multi-step and direct invocation MSIE flows. This option removes the requirement of having to specify all primary snapshots at elaboration time and allows for instance-specific partition boundaries. It also adds greater partitioning flexibility by using standard binding rules to decide which instances bind to which primaries.
  • Merging Multiple Package with same name: At the time of incremental elaboration, if the model ends up with multiple top-level packages that all have the same name but come from different file paths/partitions, the simulator throws an error IEPKGF.  To pass this error, use the -msiepkgsok option on the command line.
  • Passing Full-Design Access, Timing, or X-Propagation Files: MSIE can process access, timing, and X-propagation files that have hierarchical paths relative to the full design. Simply include the -afile, -tfile, or -xfile options on each of the xmelab or xrun command lines that build the partitions affected by these files. When using direct invocation or multi-step xrun to elaborate a primary partition that is not instantiated at the top of the full model, the -incrpath option is required. This option defines the hierarchical path from the top of the overall model to the top(s) of the partition(s) being elaborated, which is needed to tell the elaborator how to extract the relevant data.
  • Resolving Hierarchical References that Originate from a Primary: MSIE supports hierarchical references to some kinds of objects that originate in a primary partition with the -genhref option. In addition to generating an href file, this option produces a primary_externs.v file for a primary if a design unit in that primary needs access to some other part of the design or if it needs access to some shared object that is in another partition.
  • Single-Step Invocation: A single xrun command can build the complete MSIE model. This avoids the need for multiple xrun or xmelab commands when building each partition, improving usability. With single-step xrun invocation mode, the software automatically:
    • Updates primary and incremental partitions
    • Partitions Verilog configurations
    • Resolves hierarchical references
    • Elaborates all primaries with the proper parameter values
    • Elaborates primaries in parallel
    • Takes care of Access file, Xprop file, Timing file, Coverage configuration file and SDF file
  • Updating Out-of-Date Primary Snapshots Automatically: MSIE supports updating primary snapshots automatically for both xmelab and xrun:
    • Use xmelab -update when building the simulation snapshot to rebuild any out-of-date primary snapshots.

    • Use xrun to rebuild any out-of-date primary snapshots by default during incremental elaboration.

      In Multi-step MSIE flow, you can use the -noprimupdate option to disable automatic updates.

      When building an incremental snapshot for the first time, the primary is not updated if some files have changed.

      With the -forceprimupdate option, the first-time incremental checks to ensure the primary is up-to-date. Any changes in a file used during the build of the primary are detected on the build of the second (onwards) incremental and it automatically rebuilds the primary to get updated.

      Single-step MSIE flow works like monolithic flow in which automatic updates to elaboration are enabled by default and there is no way to stop disabling automatic updates to primary.
