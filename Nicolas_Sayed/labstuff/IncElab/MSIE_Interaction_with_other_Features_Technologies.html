MSIE Interaction with other Features/Technologies

This topic provides information about MSIE interaction with other features or technologies. This information applies to all supported MSIE scenarios, whether you are elaborating one or more than one primary snapshot for a given design.

Feature/TechnologyMSIE usage description

Assertion Based Verification (ABV)

ABV is supported.

If the assertion completely resides in either the incremental partition or a primary partition, the incremental elaboration works as expected. However, if the binding specified in the simulation snapshot relates to a module/instance in any of the primary snapshots which are not part of the incremental partition, then incremental elaboration does not work as expected and requires re-elaboration of the primary as well.

This is a result of SV bindings, which require a second compilation from xmelab.

Coverage

Coverage can be enabled during primary and incremental elaboration.

Functional coverage cannot be enabled at the time of generating the simulation snapshot if it was not enabled at the time of generating a primary snapshot. In addition, if functional coverage was enabled at the time of generating a primary snapshot, it automatically applies to the simulation snapshot.

Other types of coverage (block, expression, toggle, fsm) can be enabled when elaborating a primary snapshot or the simulation snapshot.

You can specify a coverage configuration file (CCF) with the -covfile option during primary or incremental elaboration. For multiple primary snapshots, all coverage specifications provided during primary elaboration must be the same for all the primaries or an error is generated during incremental elaboration.

Global settings, such as branch scoring, hit count limit, expression coverage mode (sop/control/vector), and so on, specified through CCF during primary elaboration are also applied to the incremental elaboration.

Not all coverage configuration commands are supported in the MSIE flow. An error is generated if an unsupported command is specified in the CCF file. For a complete list of supported and unsupported CCF commands in the current release, see CCF Commands Supported in the MSIE Flow in the Integrated Coverage User Guide.

HAL

Xceliumâ„¢ Simulator HDL Analysis (HAL) is supported and can be run on MSIE created snapshots.

See the HAL User Guide for more information on using HAL to analyze Verilog, VHDL, SystemC, Mixed, and e language designs.

Low-Power Simulation

For low-power simulation, incremental elaboration is limited to a dual snapshot use model. See Dual Snapshot Incremental Elaboration for Low Power IEEE 1801 or Dual Snapshot Incremental Elaboration for Low Power CPF for more details.

Mixed-Signal (MS)

Mixed-Signal is supported for logic+real number modeling designs. For other mixed-signal designs, the elaborator generates an error.

See the Incremental Elaboration for Mixed-Signal in the Spectre AMS Designer and Xcelium Simulator Mixed-Signal User Guide.

SDF Annotation

In the current release, SDF annotation is supported. You can annotate the primary or incremental partitions of a design by passing a full SDF file when elaborating each partition. However, there are some limitations. For more details, see Passing an SDF Command File to Xcelium Simulator in the SDF Timing Annotation guide.
The e Language

The e language is supported. The e code cannot be in more than one partition. It must be in either:

  • One of the primary partitions
  • In the incremental partition

If the e code is present in more than one primary, or is present in a primary and the incremental, the elaborator generates a DYNSPC error.

xrun: *E,DYNSPC: Specman was detected in multiple snapshots.

Static and dynamic linking of Cadence© Specman© Elite with Xcelium Simulator is supported. Both compiled mode and interpreted mode of e code is supported.

See Integrating the e Testbench with MSIE in the Specman Integrator’s Guide.
SV Bind Constructs

Starting Xcelium 18.10 release, the new implementation allows cross-partition SV binds for MSIE methodology which was not possible with the existing SV bind mechanism. This simply means that the new implementation allows wider deployment of MSIE for designs which uses SV binds.

Backward Compatibility

Backward Compatibility can be achieved by specifying the -oxmbind option which enables the legacy SV bind implementation. The -oxmbind option is required in elaborator options only if the user is also using -extbind option and intends to use the legacy implementation.
