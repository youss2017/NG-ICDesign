Permission Types for Objects in an href file

In an href file, each kind of reference permission to objects in the primary snapshot is represented by a single character. The permission types are as follows:

  • R–Allows the value of a net or variable, or VHDL signal or port, to be used.
  • W–Allows a Verilog variable to be written by a blocking or non-blocking procedural assignment, and allows an event to be triggered.
  • F–Allows a Verilog net or variable to be forced. Force permission implies write permission.
  • D–Allows a net to be driven by any means such as a continuous assign or connection to the output port of a primitive. Allows a variable to be driven by a continuous assignment. Drive permission implies force permission.
  • E–Allows a net, variable, or event to be used where a value change causes a process to activate. This could be an explicit event control using @ or an implicit one such as when it appears on the right-hand side of a continuous assignment. Event permission implies read permission.
  • B–Allows a disable statement to work.
  • I–Allows a task or function to be called.
  • L–Disables variable to wire optimization.

The following table shows each kind of permission, the character used to represent it in an href file, the classes of declarations for which it is relevant, and some examples of hierarchical references that would require it. The code examples refer to objects in a hypothetical scope called top.a.b. The object names (N, V, E, T, F) reflect the object kind (Net, Variable, Event, Task, Function).

Where X is a VHDL object that connects to Verilog module mod. In a mixed-language hierarchical reference, Read/Drive permission must be given to the target VHDL objects in the primary partition to set input or output mode of the Verilog module port connection.

You can use an asterisk ( * ) character to specify all relevant permissions. If no character is given, that implies an asterisk. For example:


top *         # Full permission to all objects in top (Verilog)
top ...       # Full permission to all objects in top and below
:             # Full permission to all signals and ports in VHDL top scope
: ... *       # Full permission to all signals and ports in VHDL top and below

Full permissions are automatically given to all objects in the SystemVerilog interfaces and to all objects in packages. In addition, for some cases, the tool may enable permissions and access levels that have not been explicitly requested.

Type-based href:

The syntax for type-based href is:

<target> <Object Name> <Permissions> 

  • The target can be a module name with no hierarchy delimiter '.'
  • The object can be a wire, register, function or task names or a wild character '*'.
  • The permissions here can be the current supported mnemonics like 'E', 'F', 'R', 'D', 'C' or the wild character '*'


The type-based permissions are applied to the following:

  • <module name> * <href>
    To all the objects inside all the instances of the specified module. For example: 
    module1 * RD
  • <module name> <object name> <href permission>
    To the particular object in all the instances of the specified module. For example:
    module1 reg1 EC
  • <module name> <object name> *
    To the particular object in all the instances of the specified module. For example:
    module1 reg1 *
  • <module name> * *
    To all the objects in all the instances of the specified module. For example: 
    module1 * *


Example: 

msie.sv
//msie.sv
module incr();
prim prim();
initial force incr.prim.frc_mod1.a =1'b0;
endmodule

module prim();
frc_mod frc_mod1();

frc_mod frc_mod2();
endmodule

module frc_mod();
reg a;
endmodule
//===TESTCASE ENDS ===

//href_manual.txt
frc_mod a F

xrun msie.sv -compile -clean
xrun -mkprimsnap -name prim -top prim -href href_manual.txt
xrun -top incr -primname prim

In this case the permission 'F' is applied to the register 'a' in the instances frc_mod1 and frc_mod2 of the module 'frc_mod'. No autohref file is created, as complete href permissions are provided during primary snapshot elaboration.

  • Type-based href is supported only for Verilog modules.
  • The depth feature (specified with … ) is not supported with type-based href support.
