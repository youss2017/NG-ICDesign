Product Version 23.09, September 2023

Types of MSIE Flows

Monolithic elaboration (and also re-elaboration time) can be reduced with the implementation of the MSIE flow. From the perspective of implementation, there are mainly two MSIE flows:

Type of flowDescription

Single-Step Invocation (with single xrun)

A single xrun command can build the complete MSIE model. This is the simplest form of MSIE implementation, and only requires identifying the non-changing and changing part of the design code. The non-changing block names (module or entity) need to be added with –primtop on the monolithic xrun elaboration script. Modules outside the primtop module hierarchy are considered as the changing part of the design. Names of the changing blocks can also be provided with -incrtop. There may be more than one primtop and those can be sandwiched as well. Similarly, there can be more than one incrtop as well.

Primtops are created as unique primary snapshots. The incrtop modules and the modules outside the primtop module hierarchy are placed in one incremental snapshot. After identifying the primtops (non-changing modules/entity), you can leverage the parallel elaboration of those non-changing primary snapshots with the -primparallelelab<N> option, where N is number of parallel primary. Thus, the sequential monolithic elaboration time can be cut down by parallelizing the big non-changing parts (primary snapshots) with the single-xrun MSIE flow. Hence, the total elaboration time in single-xrun MSIE comes as the longest of all parallel primary snapshots + the time taken in building the final/incremental snapshot.

The more evenly the size of non-changing parts (primary snapshots) is distributed, the better the total turnaround time is.

Thus, by just adding the -primtop and -incrtop options on the monolithic xrun elaboration script, you can have the single-xrun MSIE flow and quickly leverage the MSIE gain.

With the use of Auto-MSIE partitioner, it has become further easier to generate the partitions and use in parallel and incremental MSIE build flow.

Multi-Step Invocation (with multiple xrun or xmelab)

This flow provides granular control to the user to separate the non-changing and changing part of elaboration with multiple xrun or xmelab scripts. The user can have separate physical directory paths of the elaborated primary snapshots. They can also have different flavors of the primary snapshots (different versions of the source code, different HDL/HVL language based source code). This flow requires the -mkprimsnap option to create a separate primary snapshot. Because this flow has been made to provide more granular level control, the user may have to take some initial setup actions, such as getting the href file, extern module, -incrpath information, and parameterization information. Some of these information sets are dumped from the additional -genhref step.

The -incrbind option is used at the time of building the primary snapshot. The option identifies the name of a design unit whose instances are left unbound in the elaborated primary, to be bound to another snapshot when building the final model.With -incrbind, the -mkprimsnap option is implied and does not have to be given explicitly.
