UVM Test Modification Partitions

When partitioning the DUT, the top testbench module, and other packages that do not need frequent changes in code during the testbench development process, you can elaborate them into a primary snapshot. This primary partition can include UVM testbench packages, VIP, standard packages, packaged library code, user-defined packages, and so on.

The incremental partition may contain test packages, sequence libraries, or any other testbench code that is under development or likely to change.

This type of partitioning is particularly suitable when the DUT and a large part of the testbench (packages) are stable. Since most of the code goes into the primary partition, this partitioning style offers a potential decrease in re-elaboration time.

General UVM Guidelines

The following guidelines should be followed when partitioning the design:

  • The user classes derived from UVM in the primary portion should be defined inside a package and not inside a compilation unit scope or a module. Because these classes are defined in a package, they can be extended in the incremental partition using package references.
  • The portion of the code that needs to be in the incremental partition should be defined in a separate scope, either a package or a module.
  • The incremental portion should import UVM and other user packages that are required to extend the classes you want to extend.
  • Interfaces used in the environment are instantiated in the primary top and are passed to both the DUT top and incremental top. The interfaces should be compiled in the primary partition.
