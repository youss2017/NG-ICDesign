Wildcard Support in Module Name

Starting Xcelium 22.01-a release, you can provide wildcards instead of module_name in manually provided href files which are provided with primary elaboration.

You can give only wildcard (*) in place of the hierarchical/module name. Once the wildcard is given, the given object is looked into all the modules/interfaces/programs of the design, and permissions are applied to the given objects. This is explained in the example below:

* <object name> <href permission>

where the href syntax to support this feature is:

<target> <Object Name> <Permissions>

In the above example,

  • The target is always a wildcard
  • The object can be a wire, register, function, or task names
  • The object name cannot be a wildcard or cannot have a wildcard in it
  • The permissions here could be the currently supported mnemonics like 'E', 'F', 'R', 'D', 'C' or the wild character '*'
  • All the current limitations in the href support mechanism are applicable here as well.

Supported Formats

The list of supported formats is given below:

  • * <object name> <href permission>
    The specified href permissions are applied to all the objects of a given name anywhere in the design (in all the instances of the module having that object name). 
    The following is an example of the href entry in this case:
    * reset RD
  •  * <object name> *
    All applicable href permissions are applied to the object of a given name in all the instances of the modules having this object. The following is an example of the href entry in this case:
    * reg *

Unsupported Formats

The following formats are unsupported:

  • *     *    <href_permissions>
        Using a wildcard in place of module name and object name. For example, 
        *    *   RD
  • *    *    *
       Using a wildcard in place of module name, object name, and in place of permissions.

For these two scenarios, an error would be thrown if <module_name> and <object_name> both have a wildcard  “*”.

There are previous flows of user provided hrefs where such permissions can be given:

# Full permission to objects in dut
-href “@dut *”
Named_blocks are not supported in href.

Example

msie.sv
module incr();
prim prim();
initial force incr.prim.frc_mod1.a =1'b0;
endmodule

module prim();
frc_mod frc_mod1();

frc_mod frc_mod2();
endmodule

module frc_mod();
reg a;
endmodule
//===TESTCASE ENDS ===

//href_manual.txt

* a F

xrun msie.sv -compile -clean
xrun -mkprimsnap -name prim -top prim -href href_manual.txt
xrun -top incr -primname prim

In this case, the permission 'F' is applied to the register 'a' in both the instances frc_mod1 and frc_mod2 of the module 'frc_mod'.

The autohref file is be created as complete href permissions are provided during primary snapshot elaboration.
